# 验收清单规范

## 文件与格式

必须生成 `TEST_PLAN.md`，严格使用以下格式：

```md
## 正常情况
- [ ] <条目文字>

## 捣乱情况
- [ ] <条目文字>

## 权限情况
- [ ] <条目文字>   # 不适用也要写 NA + 原因
```

## 条目写作规则

### 必须可断言
每条必须包含：
- **输入**：触发条件或操作
- **期望输出/状态**：预期结果
- **错误码/提示**：或可观测副作用

### 禁止含糊词
❌ "应该正常工作"
❌ "合理提示用户"
❌ "正确处理错误"

✅ "用户输入空密码 → 显示'密码不能为空'"
✅ "API返回500 → 页面显示'服务器错误，请稍后重试'"

## 三类场景

### 正常情况
业务主流程的正常操作：
- 用户正确输入 → 成功结果
- 正常 API 响应 → 正确渲染

### 捣乱情况
边界、异常、错误输入：
- 格式错误（Email无@）
- 边界值（超长/空/特殊字符）
- API 失败（401/404/500/超时）
- 网络错误

### 权限情况
身份与授权：
- 有/无 Token 的行为差异
- 管理员/普通用户视图差异
- 未授权操作的拒绝

## 条目文字就是"暗号"

**关键**：条目文字将直接用作测试用例的显示名，后续 Gate 工具会逐字匹配。

因此：
- 保持文字稳定，不要随意修改
- 不要有多余空格或标点差异
- 写完后不要"润色"
