# 架构锁与契约

## Architecture Lock

每次改代码前**必须**先输出：

```
Architecture Lock
- Affected scope: <本次问题/需求一句话>
- Files & Layers:
  - <file1> -> <UI/Interface/UseCase/Domain/Infra>
  - <file2> -> <...>
- Allowed changes (<=3 files): <列表>
- Red lines (must hold):
  - Library-first
  - 禁止垃圾桶模块
  - 职责分离
- If cannot fix within boundary: STOP and write ADR-lite
```

## 架构契约（Architecture Contract）

### 核心原则

- **Clean Architecture & DDD**：业务逻辑独立于框架；用例清晰隔离；边界清晰
- **Library-First**：**ALWAYS** 先找现成方案/库，再写自定义代码
- **Separation of Concerns**：**DO NOT** 混合层级职责

### 硬红线（Invariants）

| 红线           | 说明                                    |
| -------------- | --------------------------------------- |
| 依赖方向       | 低层不得依赖高层；Domain 不得依赖 Infra |
| UI边界         | 业务规则不进 UI 组件                    |
| Controller边界 | 不直接写 DB 查询或业务规则              |
| 命名规则       | 禁止 `utils/helpers/common/shared`      |
| 无跨上下文耦合 | 不为修一点暴露另一模块内部              |

### 变更围栏

每轮修复最多 **3 个文件**。

跨层、目录重构、引入通用模块、接口大改：必须先输出 ADR-lite。

## ADR-lite

当必须跨边界时，输出：

```
ADR-lite
- Why boundary must change: <为什么必须跨边界>
- Alternatives considered: <考虑过的替代方案>
- Risks & rollback plan: <风险与回滚方案>
- Minimal plan: <最小实施计划>
```

然后**停止**，等待人工批准。
