# 救火模式

## 触发条件

失败数量很多（如 >5 个），需要系统性修复。

## 强制流程

### 1. 全量摸底

先跑全量测试收集失败概况：

```bash
pytest --tb=no -q  # 只看失败列表
```

### 2. 分组

按错误类型/模块/根因分组：

| 优先级 | 类型      | 示例                        |
| ------ | --------- | --------------------------- |
| P0     | 基础设施  | ImportError, ModuleNotFound |
| P1     | 依赖/配置 | 缺少环境变量, 配置错误      |
| P2     | API 变更  | AttributeError, 签名变更    |
| P3     | 业务断言  | AssertionError              |

### 3. 按组修复

**每次只修一组**：

```
Round 1: 修 ImportError (8个)
  → 跑该组相关子集
  → 全部绿后继续

Round 2: 修 AttributeError (5个)
  → 跑该组相关子集
  → 全部绿后继续

Round 3: 修 AssertionError (2个)
  → 跑该组相关子集
  → 全部绿后继续
```

### 4. 全量回归

所有组修完后，跑全量测试确认无新问题。

## 禁止行为

| 禁止         | 原因                     |
| ------------ | ------------------------ |
| 乱序修复     | 可能改了 P3 导致 P0 更多 |
| 跨组混修     | 难以追踪问题             |
| 不跑子集验证 | 不知道这组是否真的修好   |
| 超出变更围栏 | 扩大影响范围             |

## 仍需遵守

- Change Budget（≤3 文件）
- Architecture Lock
- 禁止越界
