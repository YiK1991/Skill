# 严格 TDD 流程

## 铁律

```
NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST
没看见红灯就写实现 = 违规
```

## 红绿循环

对 TEST_PLAN.md 的每条条目，按顺序执行：

### 1. RED：写最小 failing test

- 测试显示名必须与条目文字**逐字一致**
- 只写刚好能失败的最小代码
- 不要预测后续需求

### 2. VERIFY RED：必须亲眼看到失败

- 运行测试，确认失败
- 失败必须是"功能缺失/行为不符"
- **不能是语法错误或环境问题**

```bash
# 期望看到类似：
FAILED: Expected "密码错误" but got undefined
```

### 3. GREEN：写最小实现

- 只写刚好让当前测试通过的代码
- 不要"顺手"加其他功能
- 可以用最丑的方式实现

### 4. VERIFY GREEN：确认通过

- 重跑测试，确认通过
- 同时检查已有测试（回归）仍然绿

### 5. REFACTOR：全绿后清理

- 只在全绿状态下重构
- 每次小改动后重跑测试
- 保持绿灯

## 一次只做一条

不要批量写多个测试再批量实现。

正确流程：
```
条目1: RED → GREEN → REFACTOR
条目2: RED → GREEN → REFACTOR
条目3: RED → GREEN → REFACTOR
```

错误流程：
```
条目1,2,3: RED RED RED → GREEN GREEN GREEN # ❌
```

## 常见违规

| 违规             | 后果                   |
| ---------------- | ---------------------- |
| 没跑测试就写实现 | 不知道测试是否有效     |
| 测试"恰好"通过了 | 可能测试写错了         |
| 一次写多个测试   | 失去细粒度反馈         |
| 在红灯状态重构   | 不知道改动是否破坏功能 |
