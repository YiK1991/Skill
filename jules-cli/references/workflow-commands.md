# Jules 工作流命令参考\r\n\r\n> 从 SKILL.md 搬出的完整 CLI 命令、JSON 输出示例、方法对比和批量模板。\r\n\r\n---\r\n\r\n## 提交任务（Submit）\r\n\r\n> **标准入口**：所有提交必须经过 `dispatch_prompt_pack.py`（H4）。\r\n> 脚本自动处理：UTF-8 / ASCII 路径 / BOM 剥离 / PACK.md 过滤 / smoke test / CLI-batch 保护。\r\n\r\n### Method A — Dispatch（推荐，唯一生产入口）\r\n\r\n```bash\r\nexport JULES_API_KEY=\"your-key\"  # API 模式，session ID 精确\r\npython scripts/dispatch_prompt_pack.py \\\r\n  --pack-dir <path/to/jules_pack> \\\r\n  --repo <owner/repo> \\\r\n  --starting-branch master\r\n# GATE-1: 只投递 PACK.md pending 状态的任务\r\n# GATE-3: 第一个任务单独 smoke test，失败则中止批量\r\n# GATE-CLI-BATCH: 无 API key 时多任务默认阻断\r\n```\r\n\r\n### Method B — 直接 Jules CLI（应急 fallback）\r\n\r\n仅当 dispatch 无法运行时使用：\r\n\r\n```powershell\r\ncmd /c \"chcp 65001 >nul && type C:\\temp\\task.md | jules remote new --repo owner/repo\"\r\n# Output: Session is created. ID: 158271... URL: https://jules.google.com/session/...\r\n# ⚠ 禁止使用 --parallel flag — 导致 API 400 (see P7)\r\n# ⚠ 此方法绕过所有 GATE 门禁，仅限应急\r\n```\r\n\r\n### Method C — Bridge（仅调试用）\r\n\r\n> ⚠ `jules_bridge.py submit` 默认被硬拦截。必须 `export _JULES_DISPATCH=1` 才能使用。\r\n> **禁止用于生产提交。**\r\n\r\n```bash\r\nexport _JULES_DISPATCH=1  # 仅调试\r\nexport JULES_API_KEY=\"your-key\"\r\npython scripts/jules_bridge.py submit \\\r\n  --repo <owner/repo> \\\r\n  --prompt-file <path/to/TASK-XXX.md> \\\r\n  --title \"TASK-XXX\" \\\r\n  --json\r\n```\r\n\r\n### 提交成功信号\r\n\r\n```json\r\n{\"ok\":true,\"mode\":\"api\",\"session_id\":\"...\",\"session_url\":\"...\",\"idempotency_key\":\"...\",\"record_path\":\"...\"}\r\n```\r\n\r\n`ok=true` + 有 `session_id` 才算任务已发出。\r\n\r\n---\r\n\r\n## 轮询 / 等待\r\n\r\n```bash\r\npython scripts/jules_bridge.py status --session-id <id> --json\r\npython scripts/jules_bridge.py wait --session-id <id> --until COMPLETED,FAILED,AWAITING_PLAN_APPROVAL --json\r\n```\r\n\r\n## 审批计划\r\n\r\n当 session 进入 `AWAITING_PLAN_APPROVAL`：\r\n\r\n```bash\r\npython scripts/jules_bridge.py approve --session-id <id> --json\r\n```\r\n\r\n## 迭代修正\r\n\r\n### 首选：同 session sendMessage\r\n\r\nsession 仍处于 `AWAITING_USER_FEEDBACK` 或未终结时：\r\n\r\n```bash\r\npython scripts/jules_bridge.py send \\\r\n  --session-id <id> \\\r\n  --message-file <path/to/FU-001.md> \\\r\n  --json\r\n```\r\n\r\n### 次选：PR 评论 `@Jules`\r\n\r\nsession 已 `COMPLETED`，需在 PR 上追加修改时，在 PR 页面使用标准\"进一步修改\"模板（见 `prompt-envelope.md`）。\r\n\r\n## 获取结果（仅通过 PR）\r\n\r\n> **⚠ 禁止 `jules remote pull`。所有异步工作通过 PR 审阅。**\r\n\r\n1. 从 `status` 输出的 `pr_url` 获取 PR 链接\r\n2. 在 PR 页面进行 Code Review\r\n3. 通过 `@Jules` 评论沟通修改需求\r\n4. 审阅通过后，由人类决定合并\r\n\r\n## 计划审批策略\r\n\r\n| 任务类型                       | `requirePlanApproval` | 说明                                 |\r\n| ------------------------------ | --------------------- | ------------------------------------ |\r\n| 改代码 / 改接口 / 引入依赖     | `true`（强制）        | 必须显式审批后才执行                 |\r\n| 补测试 / 补文档 / 极小 cleanup | 可选 `false`          | Planning Critic 会审视，仍需人工审阅 |\r\n\r\n## Session 策略\r\n\r\n- 同一主题优先在**一个 session** 内迭代（sendMessage）\r\n- 仅当方面（aspect）完全独立时才开并行 session\r\n- session 已 `COMPLETED` 后，先尝试 PR 评论 `@Jules`\r\n- 若需全新上下文，创建 continuation session 并携带先前 session ID + 已确认发现\r\n
